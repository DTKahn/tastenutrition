# M3 ‚Äî Hardening & Deploy Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Deploy the Taste+ web viewer to GitHub Pages and the Worker to Cloudflare, adding calendar caching, session-expiry UX messaging, and mobile responsive polish.

**Architecture:** The Worker deploys to Cloudflare Workers (free tier); the web app deploys to GitHub Pages by serving the repo root so that `web/app.js`'s `import '../shared/ui.js'` and `web/styles.css`'s `@import '../shared/tokens.css'` resolve correctly. No build step. Calendar responses are cached at the Cloudflare edge for 5 minutes using the Cache API, keyed by (student ID + hashed session cookie) so users never see each other's data.

**Tech Stack:** Cloudflare Workers, Cloudflare Cache API (built-in, no KV), GitHub Actions, GitHub Pages, vanilla JS/CSS ES modules.

## Global Constraints

- No stored passwords; no card data; no PII committed to the repo. `m0/` is gitignored ‚Äî never touch it.
- No bundler. Shared files (`shared/ui.js`, `shared/tokens.css`, `shared/components.css`) are imported directly via relative paths.
- Worker: TypeScript, `@cloudflare/workers-types` already in `tsconfig.json` ‚Äî `ExecutionContext` is available globally, no import needed.
- `worker/tsconfig.json` has `noUnusedParameters: true` ‚Äî every parameter added to a function must be used.
- `web/config.js` sets `window.TASTE_API_BASE` ‚Äî the only file that differs between local dev and production.
- Do not touch `shared/parse.js`, `shared/order.js`, `extension/`, or `worker/src/session.ts`. Ordering mechanism is verified; this plan is web + worker infra only.

### Task 1: Session-expiry UX

When a 401 comes back from the Worker (session expired), the app silently resets to the login form with no message. This task wires a "Your session expired" message to the login error element.

**Files:**
- Modify: `web/app.js`

**Interfaces:**
- Produces: `logout(message?: string)` ‚Äî optional session message shown on login form after redirect

- [ ] **Step 1: Update `logout()` to accept and display a message**

Replace the current `logout` function and the 401 handler in `api()`. The full diff is below ‚Äî apply it exactly.

In `web/app.js`, replace:

```js

  state.token = null;
  state.calendar = null;
  state.students = [];
  state.weekStart = null;
  localStorage.removeItem('taste_token');
  renderShell();
}
```

with:

```js

  state.token = null;
  state.calendar = null;
  state.students = [];
  state.weekStart = null;
  localStorage.removeItem('taste_token');
  renderShell();
  if (sessionMsg) {
    $('loginError').textContent = sessionMsg;
    $('loginError').hidden = false;
  }
}
```

- [ ] **Step 2: Update the 401 handler in `api()` to pass the message**

In `web/app.js`, replace:

```js

```

with:

```js
  if (res.status === 401) {
    logout('Your session expired. Please sign in again.');
    throw Object.assign(new Error('session-expired'), { sessionExpired: true });
  }
```

- [ ] **Step 3: Suppress `showCalendarError` for session-expired throws**

`initSession` and `loadCalendar` both have `catch (ex)` blocks that call `showCalendarError`. After a 401, `renderShell()` has already hidden the calendar view, so calling `showCalendarError` would be a no-op anyway ‚Äî but the `ex.message` would be `'session-expired'`, which is ugly if it ever shows. Guard both callers.

In `web/app.js`, replace the `catch` in `initSession`:

```js

    if (state.token) {
      showCalendarError(ex.message);
    }
  }
```

with:

```js
  } catch (ex) {
    if (state.token && !ex.sessionExpired) {
      showCalendarError(ex.message);
    }
  }
```

In `web/app.js`, replace the `catch` in `loadCalendar`:

```js
  } catch (ex) {
    showCalendarError(ex.message);
  }
```

with:

```js
  } catch (ex) {
    if (!ex.sessionExpired) showCalendarError(ex.message);
  }
```

- [ ] **Step 4: Also clear the session error message on a fresh login attempt**

In `web/app.js`, in `onLogin`, `loginError` is already reset with `err.hidden = true` at the top of the handler. Verify this line exists:

If it's there, no change needed. (It is there in the current code.)

### Task 3: Worker calendar caching

Cache `/api/calendar` responses at the Cloudflare edge for 5 minutes, keyed by student ID + a 16-hex-char hash of the session cookie. Users with different sessions never share cached data. The Cache API is built into the Workers runtime ‚Äî no KV, no external service.

**Files:**
- Modify: `worker/src/index.ts`

**Interfaces:**
- Consumes: `ExecutionContext` (global, from `@cloudflare/workers-types`)
- No interface changes to callers ‚Äî existing `/api/calendar` behavior is preserved; caching is transparent.

- [ ] **Step 1: Add the cookie hash helper**

In `worker/src/index.ts`, insert this function before the `export default {` line:

```typescript
async function cookieHash(cookie: string): Promise<string> {
  const buf = await crypto.subtle.digest('SHA-256', new TextEncoder().encode(cookie));
  return Array.from(new Uint8Array(buf))
    .map((b) => b.toString(16).padStart(2, '0'))
    .join('')
    .slice(0, 16);
}
```

- [ ] **Step 2: Add `ctx` to the fetch handler signature**

Replace:

```typescript
  async fetch(req: Request, env: Env): Promise<Response> {
```

with:

```typescript
  async fetch(req: Request, env: Env, ctx: ExecutionContext): Promise<Response> {
```

- [ ] **Step 3: Replace the `/api/calendar` handler with a cached version**

Replace:

```typescript
      if (url.pathname === '/api/calendar' && req.method === 'GET') {
        const cookie = await requireCookie(req, env);
        const student = url.searchParams.get('student');
        if (!student) {
          return json(env, { error: 'Missing ?student=<id>.' }, 400);
        }
        return json(env, await getCalendar(cookie, student));
      }
```

with:

```typescript
      if (url.pathname === '/api/calendar' && req.method === 'GET') {
        const cookie = await requireCookie(req, env);
        const student = url.searchParams.get('student');
        if (!student) {
          return json(env, { error: 'Missing ?student=<id>.' }, 400);
        }
        const hash = await cookieHash(cookie);
        const cacheKey = new Request(
          `https://taste-cache/v1/calendar?s=${encodeURIComponent(student)}&u=${hash}`,
        );
        const cached = await caches.default.match(cacheKey);
        if (cached) {
          return new Response(cached.body, {
            status: 200,
            headers: { 'Content-Type': 'application/json', ...cors(env) },
          });
        }
        const data = await getCalendar(cookie, student);
        ctx.waitUntil(
          caches.default.put(
            cacheKey,
            new Response(JSON.stringify(data), {
              headers: {
                'Cache-Control': 'public, max-age=300',
                'Content-Type': 'application/json',
              },
            }),
          ),
        );
        return json(env, data);
      }
```

GitHub Pages will serve the repo root so that `web/app.js`'s `import '../shared/ui.js'` resolves to `https://<user>.github.io/<repo>/shared/ui.js`. The web app URL will be `https://<user>.github.io/<repo>/web/`.

- [ ] **Step 2: Create the GitHub Actions deploy workflow**

Create `.github/workflows/deploy.yml`:

```yaml
name: Deploy to GitHub Pages

on:
  push:
    branches: [main]
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: pages
  cancel-in-progress: true

jobs:
  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/configure-pages@v5
      - uses: actions/upload-pages-artifact@v3
        with:
          path: '.'
      - uses: actions/deploy-pages@v4
        id: deployment
```

The `path: '.'` deploys the repo root. `m0/` is gitignored so it is never in the repo and never deployed.
